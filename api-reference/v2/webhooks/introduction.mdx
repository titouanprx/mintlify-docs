Webhooks allow you to receive real-time notifications when events happen on your connected accounts — new messages, connection acceptances, and more.

## Delivery Modes

Webhooks support two delivery modes depending on your setup:

<CardGroup cols={2}>
  <Card title="Hosted Mode" icon="cloud">
    No server needed. Events are delivered via **SSE stream** or **polling**. Perfect for bots, scripts, and local apps.

    Create a webhook **without** a `url` to use hosted mode.
  </Card>
  <Card title="Custom Mode" icon="server">
    Events are sent as `POST` requests to your server. Requires a publicly accessible HTTPS endpoint.

    Create a webhook **with** a `url` to use custom mode.
  </Card>
</CardGroup>

| | Hosted | Custom |
|---|---|---|
| **Setup** | No server required | HTTPS endpoint required |
| **Real-time delivery** | SSE stream (`GET /stream`) | HTTP POST to your URL |
| **Event history** | 24h stored events (`GET /events`) | Fire-and-forget |
| **Best for** | Bots, scripts, local dev | Production servers |

## How It Works

<Tabs>
  <Tab title="Hosted Mode">
    <Steps>
      <Step title="Connect your account">
        Use `POST /v2/login` to connect an account and get an `account_id`.
      </Step>
      <Step title="Create a hosted webhook">
        Call `POST /v2/webhooks` **without** a `url`. You'll receive a `webhook_id`, `stream_url`, and `events_url`.
      </Step>
      <Step title="Start monitoring">
        Call `POST /v2/webhooks/{webhook_id}/start` to begin real-time monitoring.
      </Step>
      <Step title="Receive events">
        Connect to the SSE stream or poll the events endpoint.
      </Step>
    </Steps>
  </Tab>
  <Tab title="Custom Mode">
    <Steps>
      <Step title="Connect your account">
        Use `POST /v2/login` to connect an account and get an `account_id`.
      </Step>
      <Step title="Create a webhook with URL">
        Call `POST /v2/webhooks` with your server's `url`.
      </Step>
      <Step title="Start monitoring">
        Call `POST /v2/webhooks/{webhook_id}/start` to begin real-time monitoring.
      </Step>
      <Step title="Receive events">
        Events are sent as `POST` requests to your URL in real time.
      </Step>
    </Steps>
  </Tab>
</Tabs>

## Event Types

| Event Type | Description |
|---|---|
| `message_received` | A new LinkedIn message was received |
| `accepted_invitation` | One or more connection requests you sent were accepted |
| `all` | Subscribe to all event types |

<Note>
  By default, webhooks listen to **all** event types. You can filter specific events when creating the webhook.
</Note>

## Event Payload Format

All events share the same wrapper structure:

```json
{
  "account_id": "69c127c37cae0494dd827286",
  "account_name": "My Account",
  "event": {
    "type": "...",
    // event-specific fields
  },
  "timestamp": "2025-01-15T10:30:00Z"
}
```

### Message Received

Triggered in real-time when a new LinkedIn message is received. The inner event `type` is `"message"`.

```json
{
  "account_id": "69c127c37cae0494dd827286",
  "account_name": "My Account",
  "event": {
    "type": "message",
    "timestamp": "2025-01-15T10:30:00Z",
    "message_text": "Hello! I'd love to connect.",
    "sender": {
      "name": "John Doe",
      "profile_url": "https://www.linkedin.com/in/johndoe"
    },
    "conversation": {
      "entityUrn": "urn:li:messagingThread:2-abc123",
      "participants": [
        { "name": "John Doe", "profile_url": "https://www.linkedin.com/in/johndoe" }
      ],
      "groupChat": false,
      "unreadCount": 1,
      "read": false
    },
    "reply_to": null,
    "metadata": {
      "entity_urn": "urn:li:messagingMessage:abc123",
      "conversation_urn": "urn:li:messagingThread:2-abc123",
      "delivered_at": 1705312200000,
      "read_status": false
    }
  },
  "timestamp": "2025-01-15T10:30:00Z"
}
```

<Tip>
  The `reply_to` field is populated when the message is a reply to a previous message. It contains the original message text, sender name, and profile URL.
</Tip>

### Accepted Invitation

Triggered when one or more connection requests you sent are accepted. Detected by polling the connections list every ~6 hours and diffing against the previously stored snapshot. The payload batches all new connections detected in the same check.

```json
{
  "account_id": "69c127c37cae0494dd827286",
  "account_name": "My Account",
  "event": {
    "type": "accepted_invitation",
    "count": 2,
    "new_connections": [
      {
        "profile_url": "https://www.linkedin.com/in/johndoe",
        "name": "John Doe",
        "headline": "CTO at Acme Corp",
        "connected_at": "2025-01-15T10:30:00Z"
      },
      {
        "profile_url": "https://www.linkedin.com/in/janedoe",
        "name": "Jane Doe",
        "headline": "VP Engineering at Globex",
        "connected_at": "2025-01-15T10:30:00Z"
      }
    ]
  },
  "timestamp": "2025-01-15T10:30:00Z"
}
```

<Note>
  When you create a webhook subscribed to `accepted_invitation` (or `all`), the API performs an initial sync of your existing connections in the background. The first detection check then runs ~6 hours later and every 6 hours thereafter.
</Note>

### Disconnection

Triggered when the account's LinkedIn session expires or becomes invalid.

```json
{
  "account_id": "69c127c37cae0494dd827286",
  "account_name": "My Account",
  "event": {
    "type": "disconnection",
    "reason": "LinkedIn cookie invalid or expired"
  },
  "timestamp": "2025-01-15T10:30:00Z"
}
```

<Warning>
  When you receive a disconnection event, monitoring stops automatically. You need to re-authenticate the account via `POST /v2/login` and restart monitoring.
</Warning>

## Credit Usage

Webhook monitoring consumes **10 credits per day** per active account (1 credit every 144 minutes). Credits are consumed as long as monitoring is active.

- Creating, updating, or deleting webhooks costs **0 credits**
- SSE streaming and polling cost **0 credits**
- Only the monitoring itself (the connection to LinkedIn) costs credits

## Architecture

Unlike the V1 webhook system which required creating a separate "webhook account", the V2 system works with your existing accounts:

- **V2 accounts** are created via `POST /v2/login` (profiles, messages, network, etc.)
- **Webhooks** are configurations that attach to those accounts
- One account can have multiple webhooks with different event filters
- Starting/stopping monitoring is done per webhook
